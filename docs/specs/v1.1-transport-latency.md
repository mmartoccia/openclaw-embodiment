# Spec: TransportHal Latency Awareness (v1.1)

**Location:** `openclaw_embodiment/hal/base.py` (TransportHal ABC)
**Status:** Implemented

---

## Purpose

Allows the pipeline to adapt actuation timing based on actual transport lag. When BLE is slow (~50ms), the pipeline can start TTS playback before sending the display command so both reach the user simultaneously.

## Problem Solved

Without latency awareness, the pipeline fires all actuations simultaneously. On BLE, display commands arrive ~40ms after audio starts, causing a visible desync. `get_expected_latency_ms()` lets the pipeline offset commands by transport lag.

---

## API Surface

New abstract method added to `TransportHal`:

```python
class TransportHal(HALBase, ABC):
    @abstractmethod
    def get_expected_latency_ms(self) -> int:
        """Return expected one-way send latency in milliseconds.
        BLE: ~50ms, HTTP: ~10ms, LocalMLX: ~5ms."""
        ...

    def get_measured_latency_ms(self) -> Optional[int]:
        """Rolling average of last 10 send latencies. None until first send."""
        return None
```

### Expected values by transport type

| Transport | Expected (ms) | Notes |
|-----------|--------------|-------|
| BLE 5.3 (GATT) | 50 | Includes MTU negotiation overhead |
| HTTP (LAN) | 10 | TCP round-trip on 1Gbps LAN |
| WiFi TCP | 10 | Same as HTTP; no BLE overhead |
| LocalMLX | 5 | In-process; no network |
| AttachmentTransport | 100 | Subprocess spawn + file I/O |
| Simulator | 1 | In-process queue |

---

## Data Schemas

No new schemas. `get_measured_latency_ms()` returns `Optional[int]`.

---

## Integration Points

- **EmbodimentPipeline:** Query `transport.get_expected_latency_ms()` before dispatching actuations.
- **AudioOutputHal:** May start TTS `expected_latency_ms` before display command.
- **HALRegistry:** Exposes latency info in `discover()` output.

---

## Implementation Requirements

All `TransportHal` subclasses must implement `get_expected_latency_ms()`:
- `SimulatedTransport`: 1ms + rolling average tracking
- `FrameTransportHAL`: 50ms (BLE)
- `ReachyTransportHAL`: 10ms (HTTP)
- `PiBLETransport`: 50ms (BLE)
- `PiWiFiTransport`: 10ms (TCP)
- `OakDTransportHAL`: 10ms (HTTP)
- `G2TransportHAL`: 50ms (BLE)
- `Reachy2TransportHAL`: 10ms (HTTP/gRPC)
- `LocalMLXTransport`: 5ms (in-process)
- `AttachmentTransport`: 100ms (subprocess)

`get_measured_latency_ms()` is optional -- default returns `None`. Subclasses that track per-send timing (e.g., `SimulatedTransport`) override to return the rolling average.

---

## Failure Modes

| Failure | Handling |
|---------|----------|
| No sends yet | `get_measured_latency_ms()` returns `None` |
| Outlier spike | Rolling window (last 10) smooths spikes |
| Not implemented | ABC enforcement prevents instantiation |

---

## Test Cases

1. `get_expected_latency_ms()` returns int >= 0
2. `get_measured_latency_ms()` returns None before first send
3. `get_measured_latency_ms()` returns int after sends
4. Rolling window caps at 10 samples
5. `FrameTransportHAL` returns 50ms expected latency
6. `Reachy2TransportHAL` returns 10ms expected latency
