# Design Spec: Cross-Embodiment Orchestration (v2.0)

**Status:** Design only -- implementation is v2.0
**Location (planned):** `openclaw_embodiment/core/cross_embodiment.py`

---

## Purpose

Allow a single OpenClaw agent to interact with multiple simultaneous device types (glasses + robot + phone) and route agent responses to the most appropriate device based on response type and device capabilities.

## Problem Solved

Today, each `EmbodimentSDK` instance manages exactly one device. A user wearing Even G2 glasses while interacting with a Reachy Mini robot would need two separate SDK instances that don't coordinate. `CrossEmbodimentOrchestrator` unifies them under one agent, one session, and one intent routing layer.

---

## Interface Design

```python
from openclaw_embodiment.core.cross_embodiment import (
    CrossEmbodimentOrchestrator,
    DeviceHandle,
    IntentRouter,
    RoutingPolicy,
)

orch = CrossEmbodimentOrchestrator()

# Register devices
glasses = orch.register_device("even-g2", config={"ble_address": "AA:BB:CC"})
robot   = orch.register_device("reachy2", config={"host": "reachy.local"})
phone   = orch.register_device("ios-companion", config={"host": "192.168.1.50"})

# Start all devices + agent session
await orch.start()

# Route a response manually
await orch.route(response, target=glasses)

# Let routing policy decide:
await orch.route_auto(response)  # Router selects best device

# Stop all devices
await orch.stop()
```

## `DeviceHandle`

```python
@dataclass
class DeviceHandle:
    handle_id: str
    profile: str
    config: dict
    registry: HALRegistry
    sdk: EmbodimentSDK
    capabilities: List[str]
    is_active: bool
```

`capabilities` is derived from the device's HAL ABCs at registration time:
- `display`: Device has `DisplayHal`
- `audio_output`: Device has `AudioOutputHal`
- `actuator`: Device has `ActuatorHal`
- `camera`: Device has `CameraHal`
- `microphone`: Device has `MicrophoneHal`

### `RoutingPolicy`

```python
class RoutingPolicy(Enum):
    CAPABILITY_MATCH = "capability_match"
    # Route TEXT/AUDIO -> device with audio_output; DISPLAY -> device with display.
    # First matching device wins.

    BROADCAST = "broadcast"
    # Route response to ALL devices that can handle the response type.
    # Useful for alerts (show on glasses AND robot display).

    PROXIMITY = "proximity"
    # Route to physically closest device to the user.
    # Requires devices to report location or RSSI.

    EXPLICIT = "explicit"
    # Agent response metadata includes target device handle_id.
```

### Default routing table (CAPABILITY_MATCH)

| ResponseType | Preferred capability | Fallback |
|-------------|----------------------|---------|
| TEXT | audio_output | display |
| AUDIO | audio_output | none |
| DISPLAY | display | none |
| ACTION | actuator | none |
| HEARTBEAT | any | any |

## Agent Session Management

One OpenClaw session is shared across all devices. The session receives context from all devices (triggers may come from glasses OR robot OR phone). The `CrossEmbodimentOrchestrator` labels each context payload with `device_id` so the agent knows the source.

```python
@dataclass
class MultiDeviceContext:
    """Aggregated context from multiple devices."""
    timestamp_epoch: int
    sources: List[ContextPayload]  # One per contributing device
    primary_device_id: str         # Device that triggered the capture
```

---

## Conflict Resolution

When multiple devices trigger simultaneously:
1. `TriggerArbiter` deduplicates within a cross-device window (default 1000ms, wider than single-device 500ms).
2. Primary device is the one with the highest `trigger_confidence`.
3. Secondary devices' sensor data is included as supplemental context.

## Lifecycle

```
CrossEmbodimentOrchestrator.start()
  -> DeviceRegistry.register_all()
  -> EmbodimentSDK.start() x N (one per device)
  -> TriggerArbiter (global, cross-device)
  -> OpenClaw session create (one session, shared)
  -> IntentRouter.bind(session)

On trigger:
  -> Any device fires TriggerEvent
  -> Global TriggerArbiter arbitrates
  -> Context from all devices aggregated
  -> Agent processes (one response)
  -> IntentRouter routes response to N devices

CrossEmbodimentOrchestrator.stop()
  -> EmbodimentSDK.stop() x N
  -> Session cleanup
```

## Open Questions (for v2.0 implementation)

1. Should devices share a single `TriggerArbiter` or have per-device arbiters that feed a global one?
2. How does session context handle concurrent triggers from glasses and robot simultaneously?
3. PROXIMITY policy requires device location -- should this be via GPS, BLE RSSI, or manual config?
4. Should `CrossEmbodimentOrchestrator` be built on `HalOrchestrator` internally (v2.0 dependency)?
