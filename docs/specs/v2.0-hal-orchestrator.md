# Design Spec: HalOrchestrator (v2.0)

**Status:** Design only -- implementation is v2.0
**Location (planned):** `openclaw_embodiment/core/orchestrator.py`

---

## Purpose

Replace manual pipeline wiring in `EmbodimentPipeline` with a first-class `HalOrchestrator` object. The `trigger → capture → classify → transport → actuate` loop becomes introspectable, hookable, and async-native. Per-stage middleware enables logging, metrics, retries, and custom preprocessing without forking the SDK.

## Problem Solved

`EmbodimentSDK` wires the pipeline manually in `start()` and `_loop()` methods. Adding middleware (latency logging, A/B testing transports, custom classifiers) requires subclassing or monkey-patching. `HalOrchestrator` promotes the loop to a first-class object with well-defined hooks.

---

## Interface Design

```python
from openclaw_embodiment.core.orchestrator import (
    HalOrchestrator,
    OrchestratorStage,
    StageHook,
    OrchestratorConfig,
)

# Define hooks
async def log_trigger(event: TriggerEvent, ctx: StageContext) -> TriggerEvent:
    print(f"[TRIGGER] {event.event_id}")
    return event

async def latency_metric(result: SendResult, ctx: StageContext) -> SendResult:
    metrics.record("transport_ms", result.elapsed_ms)
    return result

# Configure
orchestrator = HalOrchestrator(
    registry=registry,
    config=OrchestratorConfig(
        hooks={
            OrchestratorStage.TRIGGER: [log_trigger],
            OrchestratorStage.TRANSPORT: [latency_metric],
        }
    ),
)

await orchestrator.run()  # Async main loop
```

## `OrchestratorStage`

```python
class OrchestratorStage(Enum):
    TRIGGER    = "trigger"    # IMU/audio event detection
    CAPTURE    = "capture"    # Camera frame + audio grab
    CLASSIFY   = "classify"   # Classifier gate (interesting/uninteresting)
    TRANSPORT  = "transport"  # Payload delivery to agent
    ACTUATE    = "actuate"    # Physical actuator commands
    RESPONSE   = "response"   # Agent response -> device
```

---

## `StageHook` Protocol

```python
class StageHook(Protocol[T_in, T_out]):
    async def __call__(self, data: T_in, ctx: StageContext) -> T_out:
        """Transform or observe stage data. Must return data (may be modified)."""
        ...
```

Hooks are composable: multiple hooks per stage are applied in order. A hook may raise to abort the stage.

## `OrchestratorConfig`

```python
@dataclass
class OrchestratorConfig:
    hooks: Dict[OrchestratorStage, List[StageHook]]
    polling_hz: int = 25
    concurrent_triggers: int = 1   # Max concurrent trigger streams
    error_policy: str = "continue" # "continue" | "abort" | "retry"
    retry_count: int = 3
    metrics_enabled: bool = True
```

## Key Design Decisions

### Async-native
The orchestrator runs on `asyncio`. Blocking HAL calls (camera capture, transport send) are wrapped in `asyncio.get_event_loop().run_in_executor()`. This allows concurrent trigger streams without blocking.

### Concurrent trigger streams
Multiple sensors (IMU + audio + proximity) can fire simultaneously. The orchestrator queues trigger events and processes up to `concurrent_triggers` in parallel. `TriggerArbiter` handles deduplication before events reach the orchestrator.

### HAL HAL isolation
Each HAL is called through its ABC. The orchestrator never accesses device-specific APIs directly -- all device coupling is in the HAL implementation layer.

### Hook composition
Hooks are pure functions (async). They receive stage input, may transform it, and return it. A hook that returns None aborts the stage. Hooks are ordered; earlier hooks run before later ones.

## Migration Path from v1.x

`EmbodimentSDK` (v1.x) remains supported. `HalOrchestrator` (v2.0) is a separate entrypoint:

```python
# v1.x (still works)
sdk = EmbodimentSDK(registry=registry)
sdk.start()

# v2.0 (new)
orch = HalOrchestrator(registry=registry, config=config)
await orch.run()
```
