# Spec: TriggerArbiter (v1.1)

**Location:** `openclaw_embodiment/triggers/arbiter.py`
**Status:** Implemented

---

## Purpose

Fuses `TriggerEvent` objects from multiple concurrent sources (visual `TriggerDetector`, `AudioTriggerDetector`, future IMU/proximity sensors) into a single downstream callback. Prevents double-firing when both audio and visual triggers fire for the same environmental event.

## Problem Solved

Without arbitration, a loud sound at the same moment as a saccade would fire two separate context captures within milliseconds -- sending duplicate payloads to the agent. `TriggerArbiter` deduplicates within a configurable window and applies a priority policy to select the better signal.

---

## API Surface

```python
from openclaw_embodiment.triggers.arbiter import (
    TriggerArbiter,
    ArbiterPolicy,
    ArbiterConfig,
)
from openclaw_embodiment.core.trigger import TriggerEvent

config = ArbiterConfig(
    policy=ArbiterPolicy.FIRST_WINS,
    dedup_window_ms=500,
)

def on_fused(event: TriggerEvent) -> None:
    pipeline.handle_trigger(event)

arbiter = TriggerArbiter(on_trigger=on_fused, config=config)
arbiter.push(visual_event)
arbiter.push(audio_event)  # suppressed within window
```

### Classes

#### `ArbiterPolicy`
```python
class ArbiterPolicy(Enum):
    FIRST_WINS = "first_wins"          # First event wins; suppress duplicates
    AUDIO_PRIORITY = "audio_priority"  # Audio beats visual within window
    VISUAL_PRIORITY = "visual_priority"# Visual beats audio within window
    HIGHEST_CONFIDENCE = "highest_confidence"  # Best confidence score wins
```

#### `ArbiterConfig`
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `policy` | `ArbiterPolicy` | `FIRST_WINS` | Priority policy |
| `dedup_window_ms` | `int` | `500` | Deduplication cooldown window |
| `max_pending` | `int` | `16` | Max pending events for HIGHEST_CONFIDENCE |

#### `TriggerArbiter`
| Method | Returns | Description |
|--------|---------|-------------|
| `__init__(on_trigger, config)` | -- | Construct with callback and optional config |
| `push(event)` | `None` | Submit event for arbitration (thread-safe) |

## Policy Behavior

### FIRST_WINS
- First event after the cooldown window emits immediately.
- All subsequent events within `dedup_window_ms` are dropped.
- Recommended for low-latency scenarios.

### AUDIO_PRIORITY
- Within the dedup window, if an audio event arrives after a visual event, the audio event is promoted and emitted.
- Audio event IDs must contain "audio", "snd", or "mic" (case-insensitive) for detection.

### VISUAL_PRIORITY
- Inverse of AUDIO_PRIORITY -- visual events replace audio within the window.

### HIGHEST_CONFIDENCE
- Events are accumulated for the full `dedup_window_ms` window.
- At window expiry, the event with the highest `trigger_confidence` is emitted.
- A background `threading.Timer` manages window expiry.

---

## Integration Points

- **AudioTriggerDetector:** Wire callback to `arbiter.push()`.
- **TriggerDetector:** Wire `update()` return value to `arbiter.push()`.
- **EmbodimentPipeline:** Replace direct `TriggerDetector.update()` call with `TriggerArbiter`-mediated path.

---

## Failure Modes

| Failure | Handling |
|---------|----------|
| Callback exception | Logged; arbiter continues |
| Timer fails to fire | Window accumulates without flush; next push will reset |
| Thread-safety | All `push()` calls hold `self._lock` |

---

## Test Cases

1. `FIRST_WINS`: second event within window is suppressed
2. `FIRST_WINS`: second event after window is emitted
3. `HIGHEST_CONFIDENCE`: lower-confidence events lose to higher-confidence
4. `AUDIO_PRIORITY`: audio event not suppressed when visual fired first
5. Thread-safety: concurrent `push()` calls from multiple threads
6. Empty callback list: arbiter handles gracefully
