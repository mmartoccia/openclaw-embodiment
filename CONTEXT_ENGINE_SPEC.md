# OpenClaw Embodiment — Context Engine Spec
**Status:** Draft v0.1 — March 3, 2026
**Authors:** Mike Martoccia

---

## Overview

The Context Engine is the layer between physical sensors and the LLM agent. It solves a problem nobody else has solved cleanly: how do you give an AI agent a coherent, honest picture of the physical world across arbitrary device configurations, with graceful degradation when sensors are missing, and without forcing everything through the bottleneck of natural language?

The answer is three things:
1. A typed schema (`SensorContext`) that is always valid regardless of available sensors
2. A maintained world state (`WorldModel`) with explicit staleness tracking
3. A burst output model that bypasses natural language for time-sensitive responses

---

## Architecture: Three Timescales

```
Fast   < 100ms   Signal tier       Raw sensor events → immediate device output
                                   No LLM involved. Tone, haptic, LED.

Medium  1–3s     Context tier      Sensors → SensorContext snapshot
                                   LLM receives structured context, makes typed decision
                                   → ResponseBurst dispatched to output devices

Slow    Rolling  World tier        SensorContext history → maintained WorldModel
                                   Continuously updated semantic state of environment
                                   Feeds medium tier with persistent entity knowledge
```

Each tier is independent. The fast tier never waits for the slow tier. The slow tier consumes the medium tier's output asynchronously.

## Layer 1: SensorContext (Medium Path)

The canonical context object. Always the same schema. Fields are `Optional` — `None` means "sensor unavailable", not "sensor returned nothing."

```python
@dataclass
class SensorContext:
    # Identity
    timestamp_ms: int
    device_id: str
    trigger: str              # "voice_detected" | "motion" | "ble_new_device" |
                              # "scheduled" | "manual" | "threshold_crossed"

    # Sensor channels (all Optional — graceful degradation)
    audio:     Optional[AudioContext]
    visual:    Optional[VisualContext]
    motion:    Optional[MotionContext]
    proximity: Optional[ProximityContext]

    # Fused outputs
    awareness_level: float    # 0.0–1.0: quality of correlation across available sensors
                              # NOT just sensor count — disagreeing sensors lower this
    conflicts: List[str]      # e.g. ["audio:3_speakers visual:1_person"]
    summary: str              # LLM-ready natural language, generated by fusion layer

    # Capability fingerprint (what this device CAN provide, vs. what it DID provide)
    device_capabilities: DeviceCapabilityVector
```

### Channel Schemas

```python
@dataclass
class AudioContext:
    transcript:     Optional[str]    # STT output
    speaker_count:  Optional[int]    # distinct voices detected
    ambient_class:  str              # "speech" | "music" | "silence" | "noise"
    rms_level:      float            # raw signal energy
    language:       Optional[str]    # detected language code
    confidence:     float            # 0.0–1.0

@dataclass
class VisualContext:
    description:    Optional[str]    # scene description (local VLM or cloud)
    person_count:   Optional[int]
    activity:       Optional[str]    # "sitting" | "walking" | "talking" | "idle"
    lighting:       str              # "bright" | "dim" | "dark"
    frame_path:     Optional[str]    # path to captured frame for downstream use
    confidence:     float

@dataclass
class MotionContext:
    state:          str              # "stationary" | "walking" | "running" | "gesture"
    orientation:    Optional[Tuple[float, float, float]]   # yaw, pitch, roll
    acceleration:   Optional[Tuple[float, float, float]]
    confidence:     float

@dataclass
class ProximityContext:
    known_devices:  List[str]        # names of recognized BLE devices
    unknown_count:  int              # unrecognized BLE devices in range
    rssi_map:       Dict[str, int]   # device_id → RSSI
    confidence:     float
```

### DeviceCapabilityVector

Every device registers its sensor profile at init. The context engine uses this to know what `None` means: "sensor failed" vs. "sensor doesn't exist on this device."

```python
@dataclass
class DeviceCapabilityVector:
    has_microphone:   bool
    has_camera:       bool
    has_imu:          bool
    has_ble:          bool
    has_display:      bool
    has_speaker:      bool
    has_gps:          bool
    # Wearable-specific
    is_wearable:      bool    # changes context interpretation (user-centric vs room-centric)
    has_gaze:         bool    # attention direction available
```

## Layer 2: WorldModel (Slow Path)

A continuously maintained semantic state of the environment. Fed by SensorContext history. Never the authoritative source for real-time decisions — that's the medium path's job. But always available to enrich context with persistent knowledge.

```python
@dataclass
class EntityState:
    entity_id:      str
    entity_type:    str              # "person" | "device" | "object"
    last_seen:      int              # timestamp_ms
    last_context:   str              # what was happening when last observed
    confidence:     float            # decays over time
    source_sensors: List[str]        # which sensors contributed to this state

@dataclass
class WorldModel:
    device_id:       str
    last_updated:    int             # timestamp_ms

    current_context: SensorContext   # most recent snapshot
    history:         deque           # rolling window of SensorContext (configurable depth)
    window_seconds:  int             # how many seconds of history to maintain

    entity_states:   Dict[str, EntityState]   # known entities + their states
    narrative:       str             # maintained natural language world description
    staleness:       float           # age of model in seconds since last update

    # Perspective type — shapes how context is interpreted
    perspective:     str             # "room_centric" | "user_centric" | "fused"
```

### Staleness and Confidence Decay

Entity confidence decays linearly by default (v1), configurable per entity type:

```
confidence(t) = initial_confidence × max(0, 1 - (t / half_life_seconds))
```

Default half-lives:
| Entity type | Half-life |
|---|---|
| Person (audio confirmed) | 120s |
| Person (visual only) | 60s |
| BLE device (connected) | 300s |
| BLE device (scan only) | 30s |
| Motion state | 10s |
| Ambient class | 30s |

When `confidence < 0.2`, entity moves to `last_known` state — not deleted, just demoted. The model never forgets; it becomes uncertain.

### Conflict Detection

When sensors disagree, `SensorContext.conflicts` is populated and `awareness_level` is penalized:

```
audio says: speaker_count=3
visual says: person_count=1
→ conflicts = ["speaker_count:audio(3) ≠ person_count:visual(1)"]
→ awareness_level reduced by 0.3
```

Conflicts are surfaced to the agent in the `summary` field. The agent is not expected to resolve them — it reasons under uncertainty.

## Layer 3: ResponseBurst (Output Path)

The output model. Three tiers, each bypassing the previous tier's overhead.

```
Signal  < 100ms    No LLM. Direct sensor event → device output.
Burst    100–500ms  Typed agent decision → multi-modal output. Minimal prose.
Explain  1–3s       Natural language. Only when the human needs to understand.
```

### ResponseBurst Schema

```python
@dataclass
class ResponseBurst:
    timestamp_ms:   int
    trigger_context: str             # what SensorContext triggered this

    # Output channels (all Optional — dispatched to available hardware)
    display:   Optional[DisplayCard]      # e-ink / AR display content
    audio:     Optional[AudioSignal]      # TTS OR tone/haptic (not both)
    led:       Optional[LEDSignal]        # pulse, color, pattern
    action:    Optional[DeviceAction]     # direct actuator command

    # Metadata
    burst_type: str   # "signal" | "burst" | "explain"
    urgency:    float # 0.0–1.0: shapes output selection when channels conflict
```

### AudioSignal

```python
@dataclass
class AudioSignal:
    signal_type:  str       # "speech" | "tone" | "haptic"
    content:      str       # text (for speech) or frequency/pattern (for tone)
    duration_ms:  int
    volume:       float
```

### Burst Decision Protocol

The agent receives `SensorContext` and returns `ResponseBurst`. The decision logic:

```
If awareness_level < 0.3 AND trigger == "scheduled":
    → No burst. WorldModel update only.

If trigger in ["voice_detected", "motion"] AND confidence > 0.7:
    → Signal tier. Direct output, no LLM token generation.

If context contains active question or request:
    → Explain tier. Natural language required.

Default:
    → Burst tier. Typed decision, DisplayCard + optional short audio.
```

### DeviceResponseRouter Extension

The existing `DeviceResponseRouter` dispatches by `ResponseType`. Extend to dispatch `ResponseBurst` to all available HALs simultaneously:

```python
router.route_burst(burst: ResponseBurst)
  → dispatches display to DistillerEinkDisplayHAL
  → dispatches audio to DistillerAudioOutputHAL
  → dispatches led to LEDHal (if present)
  → all in parallel, non-blocking
```

## Multi-Perspective Fusion (Room + User)

When both a fixed device and a wearable are active, the WorldModel maintains two perspective layers that are fused into a single `SensorContext`:

```
FixedDevice (Distiller):  room_centric  →  "3 people in space, ambient noise medium"
Wearable (G2):            user_centric  →  "user facing left, head tilted down"
                                       ↓
Fused SensorContext:  "3 people in room, user currently attending to 2 of them (left side)"
awareness_level: 0.82  (high — sensors corroborate, not conflict)
```

Fusion requires timestamp alignment within a configurable window (default 500ms). Sensor readings outside the window are treated as independent snapshots, not fused.

## Fiction vs. Reality (per Council review)

| Claim | Status | Notes |
|---|---|---|
| Typed schema across heterogeneous devices | ✅ Buildable now | Strong engineering pattern, no prior art in LLM-embodiment space |
| HAL validation on real hardware | ✅ Proven | Distiller CM5, BLE pendant confirmed |
| Graceful degradation via Optional fields | ✅ Buildable now | Standard pattern, just needs discipline |
| Three-timescale architecture | ✅ Plausible | Clean separation; fast path already exists (TriggerDetector) |
| Confidence decay / staleness | ⚠️ Plausible | Linear TTL is trivial; sophisticated decay requires validation |
| WorldModel entity tracking | ⚠️ Plausible | Needs bounding to prevent unbounded state growth |
| Multi-perspective fusion (room + user) | ⚠️ Plausible | Timestamp sync is the hard problem |
| Anticipatory context (pre-fetch on gaze) | ⚠️ Speculative | Requires gaze hardware + fast enough round-trip |
| Decay math as novel IP | ❌ Not novel | Kalman filters, occupancy grids — 30yr prior art |
| Semantic translation layer as IP | ✅ Defensible | No one has built the LLM-native typed context bridge specifically |

## Open Questions

1. **Summary generation at edge vs. cloud?** Edge = fast, requires local SLM. Cloud = easy, adds latency + coupling. Decision needed before v0.3.
2. **WorldModel storage backend?** In-memory (simple, volatile) vs. SQLite (persistent across restarts). Affects power consumption on wearables.
3. **Protobuf vs. JSON for SensorContext wire format?** Protobuf is faster/smaller; JSON is debuggable. Matters on battery-constrained devices.
4. **Does the WorldModel belong in the SDK or in OpenClaw gateway?** If in gateway, Pi sends raw SensorContext and gateway maintains state. If in SDK, more autonomous but more compute on edge.
